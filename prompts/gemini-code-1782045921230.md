# 📋 Brief Technique : Intégration MathJax/Markdown dans l'Éditeur Tauri-React

## 1. Objectif & Contraintes
* **Projet :** Application de prise de notes/cours (Tauri + React).
* **Priorités :** Simplicité d'usage, performance hors-ligne locale, et expérience d'écriture la plus fidèle possible à **LaTeX brut** (macros complexes, environnements, copier-coller de fichiers `.tex`).
* **Sécurité :** Respect strict des CSP (Content Security Policy) de Tauri (éviter l'injection de scripts externes ou de fonctions `unsafe-eval`).

---

## 2. Analyse des Plugins Markdown-it (Pourquoi MathJax v3)

L'évaluation des plugins `markdown-it` a conduit au rejet de la v2 au profit de l'architecture moderne de la v3 :

### ❌ `classeur/markdown-it-mathjax` (MathJax v2) — À ÉVITER
* **Méthode :** Rendu passif côté client (le plugin enveloppe le texte, un script global scanne le DOM dans le navigateur).
* **Inconvénients :** Incompatible avec les CSP strictes de Tauri, lourd à charger, provoque des clignotements graphiques (FOUC) lors des re-renders React, nécessite de manipuler les files d'attente de scripts globaux (`MathJax.Hub.Queue`).

### 📦 `@mdit/plugin-mathjax` (MathJax v3) — RECOMMANDÉ
* **Méthode :** Rendu actif synchrone (Server-Side Rendering / Compilation). Le plugin intercepte le texte brut et génère directement des **balises SVG autonomes**.
* **Avantages :** * Totalement compatible avec Tauri (zéro JS tiers exécuté dans la Webview pour le rendu).
  * Intégration parfaite dans le cycle de vie React (`dangerouslySetInnerHTML`).
  * Performance instantanée et compatibilité totale hors-ligne.
  * *Note :* Préféré à `tani/markdown-it-mathjax3` car il isole plus efficacement les blocs mathématiques contre les conflits de syntaxe Markdown (notamment les caractères spéciaux comme les crochets).

---

## 3. Autopsie du Bug Actuel (`azedit-dom.png`)

### Le Problème constaté :
Lors de l'utilisation de macros LaTeX complexes avec arguments optionnels (ex: `$$\newcommand\MN[2][n]{\mathrm{MN}_{#1}(#2)}$$`), le parseur échoue ou corrompt le rendu.

### La Cause (révélée par le DOM) :
La capture d'écran `azedit-dom.png` montre des classes de type `cm-line` et `cm-activeLine`, confirmant l'utilisation d'un éditeur basé sur **CodeMirror 6**. 
L'implémentation actuelle souffre d'un problème de flux de données :
* Au lieu de passer du texte brut à *markdown-it*, le système récupère le contenu HTML segmenté de la ligne active depuis le DOM.
* CodeMirror découpe la commande LaTeX en une suite de nœuds enfants (des `<span>` pour chaque crochet `[`, `]`, ou chiffre en raison de la coloration syntaxique active).
* *Markdown-it* reçoit un arbre HTML fragmenté au lieu de la chaîne continue originelle, brisant la détection des délimiteurs et des macros.

---

## 4. Architecture Cible & Implémentation (React)

Pour corriger ce bug, l'agent doit impérativement **séparer la donnée brute (State) de sa représentation visuelle (le DOM de l'éditeur)**.

### Étape 1 : Configurer l'instance de parsing
Utiliser le plugin moderne de la communauté pour blinder le traitement des macros TeX :

```typescript
import MarkdownIt from 'markdown-it';
import { mathjax } from '@mdit/plugin-mathjax';

export const md = new MarkdownIt({
  html: true,
  breaks: false,
}).use(mathjax, {
  output: 'svg', // Génère des tracés vectoriels isolés et sécurisés pour Tauri
  tex: {
    packages: {'[base]': [], 'ams': [], 'newcommand': []} // Support des macros standard
  }
});
```
(Voir fichiers .ts dans le même dossier)

## 5. Directives pour l'Agent Vibe-Coding

    Bannir définitivement toute fonction qui tente de parser le texte en lisant directement les nœuds HTML ou les classes .cm-line du DOM.

    Utiliser uniquement update.state.doc.toString() provenant de l'événement CodeMirror pour mettre à jour l'état de l'application.

    Remplacer le plugin actuel par @mdit/plugin-mathjax pour assurer l'immunité totale des caractères spéciaux ([ ], _, *) à l'intérieur des blocs mathématiques.

## Annexe 

Votre agent de vibe-coding a littéralement implémenté la méthode MathJax v2/v3 à l'ancienne (Côté Client) au sein d'une architecture SPA (React).

Le fait que le script tex-svg.js soit déposé manuellement dans le dossier /public et chargé via une balise <script> globale explique à 100 % le bug d'affichage et la désynchronisation avec votre éditeur CodeMirror.
Pourquoi cette approche  casse votre application

    Le problème du cycle de vie de React : Le script tex-svg.js se charge une seule fois au démarrage de l'application Tauri. Comme l'option startup: { typeset: false } est activée, MathJax attend sagement qu'on lui dise quoi faire. Le problème, c'est que lorsque vous tapez dans votre éditeur, React détruit et recrée le DOM de la prévisualisation en continu. Le script global de MathJax n'est pas au courant de ces changements et ne re-scanne pas automatiquement la page.

    Le contournement obligatoire (et instable) :
    Pour que ce script fonctionne avec React, votre agent va devoir ajouter un effet (useEffect) qui appelle manuellement la fonction globale de MathJax à chaque modification du texte :
    ```JavaScript

    window.MathJax.typesetPromise(); // Ou une file d'attente complexe
    ``` 

    C'est exactement ce qui provoque des clignotements à l'écran, des lenteurs, et des plantages dès que la formule est en cours de saisie (incomplète).

    Le conflit persistant avec Markdown-it :
    Même si le script MathJax finit par s'exécuter, le package markdown-it (s'il est configuré sans plugin mathématique) va altérer le texte (comme le découpage des crochets que l'on a vu sur votre capture d'écran du DOM) avant que MathJax n'ait le temps de lire la ligne.

### La correction à demander :

Pour obtenir une application fluide, robuste et alignée avec les standards de React et de Tauri, vous devez demander à votre agent de supprimer cette tuyauterie globale et de tout basculer dans le pipeline Node.js avec @mdit/plugin-mathjax.

Voici les instructions précises à lui donner :
1. Supprimer la configuration window.MathJax et la balise <script src="/tex-svg.js"> de votre fichier index.html.
2. Supprimer le fichier tex-svg.js du dossier /public (il devient inutile).
3. Installer le plugin via npm/yarn :
4. Initialiser Markdown-it directement dans le code de l'application (le rendu se fera de manière synchrone en JavaScript/TypeScript pur, sans dépendance globale).

Le fichier mis à jour dans le bloc de code précédent prend déjà en compte cette architecture propre. En confiant le rendu à Node, la conversion en SVG devient une simple fonction de traitement de texte, instantanée et sans aucun effet de bord dans le DOM de Tauri.
